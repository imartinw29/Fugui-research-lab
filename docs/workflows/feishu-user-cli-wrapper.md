# 在 Hermes 里保留飞书能力：为什么我最后用 lark-cli 自己接了一层

很多人折腾 Hermes 接飞书，卡住的并不是“有没有飞书”，而是一个更别扭的问题：

> 想用 Hermes 这套更顺手的执行框架，
> 但又不想失去飞书这条工作流；
> 偏偏飞书给你的不是一套开箱即用的 Hermes 集成，而是一个 CLI，再加一堆开发文档，让你自己拼。

问题不在工具少，而在**最后一公里没人替你铺好**。

所以这篇文章不想讲“怎么调用某个命令”，而是讲一个更实际的解法：

> 当现成的飞书集成不稳定、过重，或者不适合当前环境时，如何在 Hermes 里用 `lark-cli` 自己接一层可工作的用户态封装。

这不是官方最佳实践，只是一个足够实用、足够稳、而且容易维护的工程解法。

## 问题到底卡在哪里

如果你只看表面，会觉得问题很简单：

- Hermes 很好用
- 飞书有 API
- 飞书也有 CLI
- 那接一下不就行了？

真正做起来才会发现，事情不是这样。

很多时候你要的不是“调用飞书接口”这四个字，而是下面这整串能力：

- 在 Hermes 里保持统一工作流
- 还能继续用飞书做文档、搜索、知识沉淀、协作对象
- 用户态操作尽量稳定
- 不想为了某个单点工具问题，把整套环境都绑死
- 遇到 OAuth、回调、token 存储、系统依赖这些问题时，还有退路

这时你会发现：**飞书给了你零件，但没有替你装成机器。**

`lark-cli` 是零件，开发文档也是零件，但从“能调接口”到“能成为 Hermes 中稳定的一部分”，中间还差一层。

这层就是封装。

## 为什么不是直接全押在 MCP 上

如果一套现成集成已经很稳，那当然优先用现成的。

但现实里，经常会遇到这些情况：

- 用户态 token 存储不稳定
- OAuth 回调链路在某些环境下很脆
- 本地桌面、WSL、远程环境之间行为不一致
- 明明接口本身可用，但外围依赖太多

这时候问题就从“飞书能不能调用”变成了：

> **在我当前的运行环境里，哪一层更值得信任？**

我最后的判断是：对于一部分**用户态飞书操作**，直接在 Hermes 里调用 `lark-cli`，再自己包一层轻量 wrapper，反而比把所有事情都压在一个更复杂的集成链路上更稳。

不是因为 CLI 高级，而是因为它更接近一个**可控的、可替换的、足够简单的执行边界**。

## 这个封装到底解决了什么

它解决的不是“Hermes 没法连飞书”，而是更具体的事情：

### 1. 保留飞书能力，而不是放弃飞书
很多人的真实需求不是“我只要 Hermes”，而是：

- 我要 Hermes 的执行体验
- 也要飞书的协作与文档能力

那就意味着你需要一个桥，而不是二选一。

### 2. 把飞书能力降成一个可调用组件
一旦封装好了，Hermes 不需要理解飞书生态的全部复杂性，它只需要知道：

- 先检查认证状态
- 再按命令模式执行
- 错误时把 stderr / return code 带回来
- 成功时把 JSON 结果交给上层处理

换句话说，是把飞书能力从“系统难题”降成“一个工作流节点”。

### 3. 给自己留退路
封装层最大的价值之一，是可替换。

今天你可以：

- 用 `lark-cli`
- 明天也可以换成别的 wrapper
- 以后甚至可以换回更稳定的 MCP / SDK 集成

只要上层调用约定不变，你就不至于每次都把整套工作流推倒重来。

## 这层 wrapper 长什么样

核心思路其实不复杂：

1. Hermes 不直接假设自己拥有完整飞书能力。
2. 对用户态操作，优先走一个单独的 `lark-cli` wrapper。
3. wrapper 负责：
   - 拼命令
   - 继承环境变量
   - 执行子进程
   - 解析 JSON
   - 把错误转换成可读反馈
4. 上层只关心结果，不关心底层 OAuth 和 token 细节。

这样一来，Hermes 面对飞书时就不是“直接去理解一整套外部系统”，而是“调用一个已经被驯化过的小接口”。

## 最小模式

一个最小可用的模式，大概就长这样：

- `check_auth_status()`：先确认 CLI 当前是否可用
- `create_document(title, markdown)`：创建文档
- `fetch_document(doc_id)`：获取文档内容
- `search_documents(query)`：搜索文档
- `generic_api(method, path, data)`：保留一个通用出口

这几个能力加起来，已经足够支撑很多“在 Hermes 里继续用飞书”的日常场景。

## 为什么这种方式适合 Hermes

Hermes 很适合做执行与工作流编排，但并不意味着它应该把每个外部系统都吃成自己的原生器官。

相反，很多时候更好的策略是：

- 把复杂系统挡在一层清晰的边界后面
- Hermes 只接触一个足够稳定的抽象
- 外部工具的波动，不直接污染主工作流

从这个角度看，`lark-cli` wrapper 更像一种**边界管理策略**。

它承认飞书是外部系统，承认这套能力有自己的认证、状态和失败方式，然后给它一个合适的位置，而不是假装所有能力都能无缝内生化。

## 什么时候适合用这种方案

这套方案尤其适合下面这些情况：

### 场景一：你已经确定要用 Hermes 做主工作流
你不想把整个执行体验切回另一个框架，只是需要把飞书继续接住。

### 场景二：飞书操作以用户态为主
比如：

- 文档创建
- 文档抓取
- 搜索文档
- Wiki / Base 的一部分用户态操作

### 场景三：你需要一个足够轻、足够稳、可替换的层
不是最完美，但要够实用。

## 这套方法不解决什么

也要说清楚，它不是万能药。

它不解决：

- 所有飞书能力的全面封装
- 复杂权限体系的一次性打通
- 所有企业环境下的统一兼容性问题
- 任何情况下都比官方 SDK / MCP 更好

它解决的是一个更具体的问题：

> **在“想继续用 Hermes”与“又不能放弃飞书能力”之间，找一个足够工程化的中间层。**

## 我更看重它的哪一点

不是“终于能调飞书了”，而是：

> **它把一个原本容易变成环境泥潭的问题，收缩成了一个可以独立维护的接口层。**

一旦问题被收缩到这个层级，后面很多事情都简单了：

- 能测试
- 能替换
- 能记录
- 能公开分享
- 能从一次性折腾变成可复用经验

这才是最值钱的部分。

## 公开分享时应该注意什么

如果你也想把自己的这套方法公开出来，最好不要直接上传你本地的私有 skill 原件。

更稳的方式是：

- 公开思路
- 公开最小封装模式
- 公开脱敏示例脚本
- 不公开真实 token、资源 ID、路径和环境细节

也就是说，分享的是**方法和边界设计**，不是把你自己的飞书环境原样掀给别人看。

## 最后一句

飞书只给 CLI 和开发文档，让你自己搞，这件事本身并不优雅。

但换个角度看，它也迫使你想清楚一个更底层的问题：

> 到底什么叫“集成完成”？

不是接口能调通就算完成，而是这条能力真正进入了你的主工作流，而且在出问题时你知道该在哪一层修。

从这个标准看，用 `lark-cli` 在 Hermes 里自己接一层，并不是退而求其次，反而是一次很典型的工程化选择：

- 接受现实边界
- 控制复杂度
- 给系统留退路
- 把零散能力变成可维护组件

这通常比表面上的“完全集成”更可靠。
